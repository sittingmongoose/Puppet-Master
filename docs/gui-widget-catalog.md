## Curated Guidance

- `selectable_text_field`: Canonical read-only/selectable text path. Use this for selectable values in views.
- `context_menu_actions`: Canonical vertical context menu (Copy, Paste, Select all, Close).
- `auth_status_chip`: Canonical authenticated/not-authenticated badge styling.
- `page_header`: Canonical page title + right-side actions row.
- `refresh_button`: Canonical refresh action with standardized label casing and spacing.
- `responsive_form_row`: Use for config-style label + input rows (stacked on mobile, row on desktop).
- `responsive_label_value`: Use for read-only details views (ledger, history, evidence, settings).
- **Tooltip verbosity (expert vs ELI5)**: The source is Settings > Help (`settings_interaction_mode`). Use `interaction_mode_to_variant(settings_interaction_mode)` when building tooltips.

## Use This When

- Need right-click context-menu actions with copy/paste/select-all:
  use `context_menu_actions`.
- Need selectable text that supports right-click actions:
  use `selectable_text_field`.
- Need auth state badge chips in login/auth views:
  use `auth_status_chip`.
- Need top-of-page title + actions (especially refresh):
  use `page_header` with `refresh_button`.
- Need responsive label + input form rows (e.g. config tabs):
  use `responsive_form_row`.
- Need responsive label + value rows for read-only details:
  use `responsive_label_value`.
- Need tooltip variant (expert vs ELI5) for help text:
  use `interaction_mode_to_variant(settings_interaction_mode)`; the source is Settings > Help.

## Anti-Patterns

- Rebuilding copy/paste/select-all rows inline in multiple views.
- Re-implementing authenticated/not-authenticated chip styles inline.
- Hand-rolling title + refresh header rows where `page_header` fits.
- Introducing alternative selectable text primitives for normal read-only fields.
- Hardcoding platform data (models, CLI commands, auth, capabilities) — use `platform_specs` instead.
- Duplicating effort/reasoning/image support checks — use `platform_specs::supports_effort()` etc.

## DRY Data Sources (Non-Widget)

Beyond widgets, these modules are single sources of truth for platform data. **Always use these instead of hardcoding values.**

### `src/platforms/platform_specs.rs` — Platform CLI Specifications
The canonical source of truth for ALL platform CLI data. Grep for `DRY:DATA:PLATFORM_SPECS` or `DRY:FN:` in the file.

| Function | Purpose |
|----------|---------|
| `get_spec(platform)` | Full spec struct for a platform |
| `cli_binary_names(platform)` | Binary names to search for (e.g., `["agent"]` for Cursor) |
| `default_install_paths(platform)` | Where to look for CLI binaries |
| `install_methods_for(platform, os)` | Install commands filtered by OS |
| `fallback_model_ids(platform)` | Fallback models when dynamic discovery unavailable |
| `default_model_for(platform)` | Default model alias |
| `supports_effort(platform)` | Has effort/reasoning levels? (true: Claude/Codex/Copilot, false: Gemini/Cursor) |
| `effort_levels_for(platform)` | Platform-specific effort level IDs and display names |
| `supports_images(platform)` | Supports images? (ALL 5 platforms: yes) |
| `supports_plan_mode(platform)` | Supports plan mode? (ALL 5: yes) |
| `reasoning_is_model_based(platform)` | Reasoning in model names? (true: Cursor only) |
| `has_auto_mode(platform)` | Has auto model selection? (true: Cursor only) |
| `supports_subagents(platform)` | Has multi-agent? (Claude Teams, Copilot Fleets, Codex sub-agents) |
| `has_sdk(platform)` | Has programmatic SDK? (Copilot SDK, Codex SDK) |
| `image_capable_platforms()` | All platforms that support images (all 5) |
| `subagent_capable_platforms()` | Platforms with multi-agent support |

**Use this when:**
- Need platform binary names → `platform_specs::cli_binary_names(platform)`
- Need install commands → `platform_specs::install_methods_for(platform, "linux")`
- Need to check if platform supports effort → `platform_specs::supports_effort(platform)`
- Need model list → `platform_specs::fallback_model_ids(platform)` (or dynamic cache)
- Need auth info → `platform_specs::get_spec(platform).auth`

## DRY Tag Convention

All reusable code is tagged with `// DRY:` comments for grep discoverability:

```
// DRY:WIDGET:<name>  — Reusable UI widget (see src/widgets/)
// DRY:DATA:<name>    — Single source of truth data module
// DRY:FN:<name>      — Reusable helper/query function
// DRY:HELPER:<name>  — Shared utility function
```

To find all DRY-tagged items: `grep -r "DRY:" puppet-master-rs/src/`

To find widgets: `grep -r "DRY:WIDGET" puppet-master-rs/src/widgets/`
To find data sources: `grep -r "DRY:DATA" puppet-master-rs/src/`
To find functions: `grep -r "DRY:FN" puppet-master-rs/src/`
