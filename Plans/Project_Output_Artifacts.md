# Puppet Master User-Project Output Artifacts (SSOT)

## Purpose

This document defines the canonical output artifacts that Puppet Master must produce for user projects after Interviewer/Wizard planning. It is the single source of truth for filesystem layout, shard contracts, deterministic node IDs, validation pointers, and seglog persistence.

Scope is user projects in a user workspace. These outputs do not assume a `Plans/` folder in user projects.

## SSOT references (DRY)
- Locked decisions + forbidden deps: `Plans/Spec_Lock.json`
- Deterministic ambiguity handling: `Plans/Decision_Policy.md` + `Plans/auto_decisions.jsonl`
- Project plan graph index schema: `Plans/project_plan_graph_index.schema.json` (`pm.project-plan-graph-index.v1`)
- Project plan node shard schema: `Plans/project_plan_node.schema.json` (`pm.project-plan-node.v1`)
- Change budget primitive (reused by node shards): `Plans/change_budget.schema.json` (`pm.change_budget.schema.v1`)
- Evidence bundle primitive (reused by node shards): `Plans/evidence.schema.json` (`pm.evidence.schema.v1`)
- Project contract pack index schema: `Plans/contracts_index.schema.json` (`pm.project_contracts_index.schema.v1`)
- Acceptance manifest schema: `Plans/acceptance_manifest.schema.json` (`pm.acceptance_manifest.schema.v1`)
- Auto decisions JSONL row schema (reused by user projects): `Plans/auto_decisions.schema.json` (`pm.auto_decisions.schema.v1`)

## Required Output Tree

All artifacts below are required unless explicitly marked optional:

```text
.puppet-master/project/
  requirements.md
  contracts/
    index.json
  plan.md
  glossary.md             # optional (recommended)
  plan_graph/
    index.json
    nodes/
      <node_id>.json
    edges.json            # optional
  acceptance_manifest.json
  auto_decisions.jsonl
  evidence/
    <node_id>.json        # produced during execution (one per completed node)
  review/
    findings/
      <run_id>.json
    approvals/
      <run_id>.json
    checkpoints/
      <document_id>/
        <checkpoint_id>.json
```

### Artifact Roles

- `.puppet-master/project/requirements.md`: canonical requirements input for downstream execution.
- `.puppet-master/project/contracts/`: contract pack (domain and execution contracts used by graph nodes), including required `contracts/index.json`.
- `.puppet-master/project/plan.md`: human-readable plan view.
- `.puppet-master/project/glossary.md` (optional): project-specific glossary and naming conventions (recommended).
- `.puppet-master/project/plan_graph/`: machine-readable sharded execution graph.
- `.puppet-master/project/plan_graph/index.json`: sharded graph entrypoint, node listing, and validation pointers.
- `.puppet-master/project/plan_graph/nodes/<node_id>.json`: one shard per execution node.
- `.puppet-master/project/acceptance_manifest.json`: project-level acceptance index.
- `.puppet-master/project/auto_decisions.jsonl`: deterministic machine-consumable decision log.
- `.puppet-master/project/evidence/<node_id>.json`: evidence bundle produced when a node completes (schema `pm.evidence.schema.v1`).
- `.puppet-master/project/review/findings/<run_id>.json`: persisted Multi-Pass findings summary for requirements/interview review runs.
- `.puppet-master/project/review/approvals/<run_id>.json`: persisted final approval-gate decision state for review runs.
- `.puppet-master/project/review/checkpoints/<document_id>/<checkpoint_id>.json`: restorable document checkpoint records used by review/edit recovery flows.

Staging inputs may exist under `.puppet-master/requirements/*`, but canonical user-project outputs must be under `.puppet-master/project/*`.

**Execution limits are not plan artifacts:** Max concurrent execution limits (per-platform concurrency caps) are NOT part of the plan graph or node contracts. They are an execution/config concern applied by the executor from app settings (see `Plans/FinalGUISpec.md` §7.4.7). The plan graph defines only dependency and parallelism structure (`depends_on`, `parallel_group`, `blockers`/`unblocks`).

## Contract layers (Platform vs Project)

This feature defines two distinct contract layers:

1. **Platform contracts (internal SSOT):** Puppet Master’s canonical event model, tool IDs/policy semantics, and provider contracts (e.g. `Plans/Contracts_V0.md`, `Plans/Tools.md`, `Plans/CLI_Bridged_Providers.md`). User-project artifacts may cite these contracts by name/ID, but MUST NOT embed their full content.
2. **Project contracts (per user project):** The Project Contract Pack under `.puppet-master/project/contracts/` (defined below). These contracts are canonical for that project and are referenced by plan nodes via stable `ProjectContract:*` IDs.

## Project Contract Pack Contract (`contracts/`)

The **Project Contract Pack** is the project-specific contract layer generated by the Interviewer/Wizard. It defines stable interfaces and boundaries for autonomous execution.

### Required: `contracts/index.json`

- Path: `.puppet-master/project/contracts/index.json`
- Schema: `Plans/contracts_index.schema.json` (`pm.project_contracts_index.schema.v1`)
- Purpose: stable mapping from **project contract IDs** → `{ kind, path, sha256, anchors }` so plan nodes can reference contracts by stable IDs, not raw file paths.

ContractRef: SchemaID:contracts_index.schema.json

### Project contract IDs (stable)

- Format: `ProjectContract:<namespace>` with optional fragment suffix (e.g. `ProjectContract:api.accounts#/paths/~1accounts/post`, `ProjectContract:errors.taxonomy#E_AUTH_001`).
- Every plan node MUST reference **at least one** `ProjectContract:*` entry in `contract_refs`.
- Platform contracts (tool policy, event envelopes, decision policy) MUST be referenced by name/ID (e.g. `SchemaID:*`, `PolicyRule:*`) and MUST NOT be embedded verbatim into the project contract pack.

### Required contract categories (minimum coverage)

The contract pack MUST include stable contracts covering (at minimum):
- **APIs / interfaces** (endpoints, RPC, message contracts)
- **Data models / schemas** (validation + persistence shapes)
- **Module boundaries** (ownership, responsibilities)
- **Filesystem / layout** (directory structure + invariants)
- **Commands** (build, test, lint, run; deterministic)
- **Test contracts** (test suites, required fixtures, how to run)
- **Error taxonomy** (stable error codes/categories + expectations)
- **Integration boundaries** (external services, auth, rate limits, data ownership)

Chunking: contract pack content may be split across multiple files under `contracts/` as needed; `contracts/index.json` is the single required index.

## Acceptance Manifest Contract

`acceptance_manifest.json` is the project-level index of executable acceptance checks for the plan graph.

- Path: `.puppet-master/project/acceptance_manifest.json`
- Schema: `Plans/acceptance_manifest.schema.json` (`pm.acceptance_manifest.schema.v1`)
- Requirements:
  - Every node shard `acceptance[].check_id` MUST appear in the acceptance manifest for that node.
  - Checks MUST be machine-executable (no manual verification steps).
  - Checks SHOULD reference relevant `ProjectContract:*` IDs (and may also cite platform contracts like `PolicyRule:*`).

ContractRef: SchemaID:acceptance_manifest.schema.json

## Evidence Outputs (per node)

Evidence bundles are produced during orchestrator execution (not during Interviewer planning), but are part of the canonical `.puppet-master/project/` output surface.

- Path: `.puppet-master/project/evidence/<node_id>.json`
- Schema: `Plans/evidence.schema.json` (`pm.evidence.schema.v1`)
- The node shard’s `evidence_required.path` MUST point to this evidence bundle (relative to the node shard).
- Change budget enforcement applies to user-workspace code paths; Puppet Master-controlled artifacts under `.puppet-master/project/**` (including evidence + auto decisions) are excluded from budget counting.

## Review lifecycle artifacts (requirements and interview)

Review flows add canonical artifacts under `.puppet-master/project/review/`:

- `review/findings/<run_id>.json` (`review_findings_summary.v1`)
- `review/approvals/<run_id>.json` (`review_approval_gate.v1`)
- `review/checkpoints/<document_id>/<checkpoint_id>.json` (`document_checkpoint.v1`)

### Findings summary contract (`review_findings_summary.v1`)

Required fields:
- `run_id`
- `scope` (`requirements` | `interview`)
- `gaps`
- `consistency_issues`
- `missing_information`
- `applied_changes_summary`
- `unresolved_items`

### Final approval gate contract (`review_approval_gate.v1`)

Required fields:
- `run_id`
- `decision` (`accept` | `reject` | `edit`)
- `decision_timestamp`
- `decision_actor`
- `preconditions` (`findings_summary_shown=true`)

### Document checkpoint contract (`document_checkpoint.v1`)

Required fields:
- `checkpoint_id`
- `document_id`
- `label` (for example `before_multi_pass`, `after_user_edit_1`)
- `artifact_ref`
- `created_at`
- `restorable` (must be `true` for restorable checkpoints)

### Tri-location pointer contract (review surfaces)

For each review run, findings and approval artifacts must include pointers that support the three review surfaces:
1. File Editor open target (canonical path),
2. Chat pointer (same canonical path),
3. Embedded document pane reference (`document_id` and optional `selected_view`).

When evidence is available, include the evidence artifact pointer used for that run context.

## Human-Readable Plan Requirement

`plan.md` is mandatory and must summarize:

- graph entrypoints
- execution-order hints
- acceptance overview
- references to contract-pack artifacts in `.puppet-master/project/contracts/`

## Sharded Plan Graph Contract

Sharded graph output is the default for user projects.

### `plan_graph/index.json` (required)

Required top-level keys:

- `schema_version`
- `graph_id`
- `nodes`
- `entrypoints`
- `validation`

`nodes` entries must include:

- `node_id`
- `path`
- `sha256`

Optional node entry key:

- `subgraph`

`validation` must include:

- pointer to index schema ID
- pointer to node schema ID
- artifact validation targets, including:
  - acceptance manifest pointer
  - contracts index pointer

Example:

```json
{
  "$schema": "https://puppet-master.dev/schemas/project-plan-graph-index.schema.json",
  "schema_version": "1.0.0",
  "graph_id": "pmg_2026_02_23_9d4f7e",
  "nodes": [
    {
      "node_id": "ph1_t2_st1_it1_6d9c3a1f",
      "path": "nodes/ph1_t2_st1_it1_6d9c3a1f.json",
      "sha256": "5f00d8f6d4a3b5d0d6f4ab2ad0f339f8bf8f6f1702ee9be7e6f1dc5d24a0d6a9",
      "subgraph": "phase_1"
    }
  ],
  "entrypoints": ["ph1_t2_st1_it1_6d9c3a1f"],
  "validation": {
    "index_schema_id": "pm.project-plan-graph-index.v1",
    "node_schema_id": "pm.project-plan-node.v1",
    "targets": {
      "acceptance_manifest": "../acceptance_manifest.json",
      "contracts_index": "../contracts/index.json"
    }
  }
}
```

ContractRef: SchemaID:project_plan_graph_index.schema.json, SchemaID:project_plan_node.schema.json

### `plan_graph/nodes/<node_id>.json` (required per node)

Each node shard must contain all required fields.

**DRY rule:** Node shard fields reuse the same primitives as the self-build plan node contract (acceptance commands/events, evidence pointer, change budget) and MUST embed:
- `change_budget` conforming to `Plans/change_budget.schema.json` (`pm.change_budget.schema.v1`)
- `evidence_required` pointing to an evidence bundle conforming to `Plans/evidence.schema.json` (`pm.evidence.schema.v1`)
- At least one project contract reference in `contract_refs` (`ProjectContract:*`; resolvable via `contracts/index.json`)

ContractRef: SchemaID:change_budget.schema.json, SchemaID:evidence.schema.json

- `node_id`
- `objective`
- `contract_refs`
- `acceptance`
- `evidence_required`
- `allowed_tools`
- `tool_policy_mode`
- `policy_mode`
- `change_budget`
- `blockers`
- `unblocks`

Additional rules:

- `node_id` in file content must exactly match `<node_id>` in filename.
- Node IDs must be deterministic and stable (see deterministic ID rules).

Example:

```json
{
  "$schema": "https://puppet-master.dev/schemas/project-plan-node.schema.json",
  "node_id": "ph1_t2_st1_it1_6d9c3a1f",
  "objective": "Generate API skeleton for account creation endpoint",
  "contract_refs": [
    "ProjectContract:api.account#/paths/~1accounts/post",
    "ProjectContract:security.authz#account_creation"
  ],
  "acceptance": [
    {
      "check_id": "check.http.accounts.post",
      "commands": [
        {
          "cmd": "cargo test account_create -- --nocapture",
          "expect_exit_code": 0
        }
      ],
      "expected_events": ["tool.invoked"]
    }
  ],
  "evidence_required": {
    "schema_id": "pm.evidence.schema.v1",
    "path": "../evidence/ph1_t2_st1_it1_6d9c3a1f.json"
  },
  "allowed_tools": ["read", "edit", "bash"],
  "tool_policy_mode": "auto",
  "policy_mode": "workspace-write",
  "change_budget": {
    "schema_id": "pm.change_budget.schema.v1",
    "allowed_paths": ["src/**", "tests/**"],
    "forbidden_paths": [".puppet-master/**"],
    "forbidden_files": [],
    "max_files_changed": 8,
    "max_lines_added": 450,
    "max_lines_deleted": 200
  },
  "blockers": ["ph1_t1_st3_it1_49c91ab2"],
  "unblocks": ["ph1_t3_st1_it1_77ec01a5"]
}
```

### `plan_graph/edges.json` (optional)

`edges.json` is optional when edges are derivable from node-level fields (`blockers`, `unblocks`, or dependency metadata). If present, it must agree with node shard dependency semantics.

## Deterministic Node ID Rules

Node IDs must be stable and deterministic.

Rules:

- Primary source: stable lineage tuple `(phase_id, task_id, subtask_id, iteration_id)`.
- Canonical string format: normalize IDs and separators before hashing.
- Deterministic fallback: when lineage is incomplete, hash normalized stable fields (objective seed + parent lineage + deterministic ordinal) with a fixed algorithm.
- No random UUIDs for node IDs.
- Same inputs must produce identical node IDs across reruns.

Reference pattern:

```text
<phase>_<task>_<subtask>_<iteration>_<hash8>
```

## Auto-Decision Log Contract

`auto_decisions.jsonl` is newline-delimited JSON and must be deterministic and machine-consumable.

**Canonical schema:** `Plans/auto_decisions.schema.json` (`pm.auto_decisions.schema.v1`).
ContractRef: SchemaID:auto_decisions.schema.json

Example line:

```json
{"schema_id":"pm.auto_decisions.schema.v1","decision_id":"EXAMPLE.dec_ph1_t2_01","timestamp_utc":"2026-02-23T18:45:12Z","scope":"node","inputs_hash":"9c0b4f0000000000","decision":"set_policy_mode_workspace_write","rationale":"node requires file edits and tests","applied_to":[".puppet-master/project/plan_graph/nodes/ph1_t2_st1_it1_6d9c3a1f.json"],"contract_refs":["SchemaID:auto_decisions.schema.json","PolicyRule:Decision_Policy.md§2"],"example":true}
```

## Seglog Canonical Persistence Contract

All required user-project artifacts are canonical in seglog.

Requirements:

- Persist full artifact content in seglog artifact events.
- Include logical path and `sha256` for each artifact.
- Include artifact type labels for review lifecycle artifacts:
  - `review_findings_summary`
  - `review_approval_gate`
  - `document_checkpoint`
- Large artifacts use deterministic chunking with:
  - `chunk_index`
  - `chunk_count`
- Emit final integrity event per artifact with canonical `sha256` over reconstructed bytes.

Filesystem behavior:

- Files under `.puppet-master/project/*` are materializations/cache.
- Materialized files must be reproducible from seglog events.
- Replay/rebuild from seglog must reconstruct byte-identical artifacts and hashes.

## Validation Rules

Validation must enforce:

- required artifact presence in `.puppet-master/project/*`
- index required keys (`schema_version`, `graph_id`, `nodes`, `entrypoints`, `validation`)
- node shard required fields
- node filename and in-file `node_id` consistency
- deterministic ID policy conformance (no random IDs)
- `contracts/index.json` exists and validates (`pm.project_contracts_index.schema.v1`); every referenced `ProjectContract:*` resolves via the index
- `acceptance_manifest.json` exists and validates (`pm.acceptance_manifest.schema.v1`); every `acceptance[].check_id` from node shards exists in the manifest
- shard hash integrity (`index.nodes[].sha256` matches actual node file)
- node `tool_policy_mode` is present and valid (`auto` | `ask` | `deny`)
- node `evidence_required` pointer is well-formed and points under `.puppet-master/project/evidence/` (runtime gate checks evidence file existence after execution)
- when a review run exists, `review/findings/<run_id>.json` and `review/approvals/<run_id>.json` both exist and validate against required field sets above
- every approval artifact must include `preconditions.findings_summary_shown=true`
- checkpoint artifacts under `review/checkpoints/**` must be restorable (`restorable=true`) and reference a valid artifact path
- review findings/approval artifacts must include tri-location pointers (editor path, chat path, document-pane reference)
